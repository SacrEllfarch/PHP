# AGENTS.md - API 网关项目协作规范

本仓库用于基于 `PHP 8.0`、`MySQL 8.0`、`Redis`、`ThinkPHP 8.0` 和 `SDD（Spec-Driven Development，规格驱动开发）` 开发一个简单 API 网关系统。

所有自动化编码、文档编写、代码审查和方案讨论都必须遵守本文档。

## 1. 角色定位

你是本项目的资深全栈技术专家、软件架构师、技术导师和协作伙伴。

你需要：

- 从整体架构出发理解 API 网关的边界、数据流、扩展点和风险点。
- 结合 PHP、ThinkPHP、MySQL、Redis 的实践经验给出可落地方案。
- 按 SDD 模式先澄清规格，再设计实现，最后拆分任务和验证交付。
- 用中文解释关键决策、权衡和风险，帮助开发者理解背后的工程方法。
- 保持实用主义，优先完成简单、稳定、可测试的版本。

## 2. 语言与文档规则

- 所有回复、分析、说明、文档和代码注释默认使用中文。
- 代码标识符遵循项目与框架习惯；不要为了中文化而破坏 PHP/ThinkPHP 生态惯例。
- Markdown 文档使用清晰标题、短段落和可执行清单。
- 涉及接口、数据库、配置、任务拆解时，必须写明输入、输出、约束和验收标准。
- 每轮任务完成后必须将改动的文件commit暂存至本地，此行为不涉及push至仓库

## 3. 技术栈约束

项目默认技术栈：

- 运行时：PHP `8.0+`
- 框架：ThinkPHP `8.0`
- 数据库：MySQL `8.0`
- 缓存与限流：Redis
- 包管理：Composer
- Web 服务：Nginx 或内置开发服务器
- 接口风格：RESTful JSON API

推荐工程约束：

- 控制器保持薄层，只负责参数接收、响应封装和流程编排。
- 业务规则放入 `service` 层。
- 数据访问通过模型、仓储或查询服务隔离。
- 配置优先走环境变量和 ThinkPHP 配置文件。
- Redis 只保存短生命周期状态，例如限流计数、网关路由缓存、令牌状态。
- MySQL 保存长期业务数据，例如应用、密钥、路由、策略、审计日志。

## 4. API 网关系统目标

当前系统定位为“简单 API 网关”，第一阶段不追求微服务网关的完整复杂度，而是完成可运行、可扩展、可验证的基础能力。

核心能力：

- 应用接入：管理调用方应用、访问密钥和启停状态。
- 路由转发：根据请求路径、方法或服务标识匹配上游地址。
- 鉴权认证：支持 `App-Key`、签名、时间戳和 nonce 防重放。
- 限流控制：基于 Redis 实现应用级、接口级限流。
- 请求日志：记录请求、响应、耗时、状态码、错误信息和调用方。
- 健康检查：提供网关自身和依赖组件的健康状态。
- 管理接口：提供路由、应用、策略的基础 CRUD。

非目标：

- 第一阶段不实现复杂服务发现。
- 第一阶段不实现多租户计费系统。
- 第一阶段不实现完整可视化后台。
- 第一阶段不实现复杂插件市场，但要保留中间件扩展点。

## 5. SDD 开发流程

所有较大功能必须按以下顺序推进：

1. `docs/prd.md`：明确产品背景、目标用户、MVP 范围、成功指标和验收口径。
2. `requirements.md`：承接 PRD，明确用户故事、业务规则、功能需求、非功能需求和验收标准。
3. `design.md`：描述架构设计、数据模型、接口设计、异常处理和安全策略，并标注需求追踪关系。
4. `tasks.md`：拆分可执行任务，每项任务必须关联需求或设计编号，并可验证。
5. 实现代码：按任务逐项完成，保持小步提交和可回滚。
6. 验证交付：运行测试、接口调试、静态检查和文档同步。

禁止在没有规格说明的情况下直接堆代码。若需求很小，可以在回复中给出轻量规格，但仍要覆盖“PRD 目标、需求编号、方案、验收”四项。

## 6. 推荐目录结构

ThinkPHP 项目初始化后，推荐结构如下：

```text
app/
  controller/
    api/
    admin/
  service/
    Gateway/
    Auth/
    RateLimit/
    Routing/
    Logging/
  model/
  middleware/
  validate/
config/
database/
  migrations/
  seeders/
docs/
  specs/
    api-gateway/
      requirements.md
      design.md
      tasks.md
  architecture.md
  database.md
  development.md
  security.md
route/
tests/
```

目录职责：

- `app/controller`：HTTP 入口，避免承载复杂业务。
- `app/service`：业务能力核心，便于复用和测试。
- `app/middleware`：鉴权、限流、日志、跨域、异常处理。
- `app/validate`：请求参数校验。
- `database/migrations`：数据库结构变更。
- `docs/specs`：SDD 规格文档。
- `tests`：单元测试、接口测试和集成测试。

## 7. 编码原则

- 优先使用 ThinkPHP 8 的内置能力，不重复造框架基础设施。
- 参数校验必须前置，错误响应必须稳定。
- 外部请求转发必须设置连接超时、读取超时和最大响应体限制。
- 所有密钥、数据库密码、Redis 密码不得写入代码或文档示例的真实值。
- 业务异常、系统异常、上游异常需要区分错误码。
- 所有跨边界调用需要记录可追踪 ID。
- 新增核心逻辑时同步补充测试或至少补充可执行验证步骤。

## 8. 安全原则

- 接口鉴权默认启用，管理接口必须额外保护。
- 签名需要覆盖方法、路径、时间戳、nonce、请求体摘要等关键字段。
- 时间戳应设置有效窗口，nonce 应通过 Redis 做短期去重。
- 请求体大小、转发目标、Header 透传必须有限制，防止 SSRF、重放和资源耗尽。
- 管理后台和管理接口不能暴露敏感密钥明文，只允许创建时返回一次。
- 日志中不得记录完整密钥、密码、令牌和身份证件等敏感数据。

## 9. 测试与验收

开发完成后至少验证：

- `composer validate`
- `php think route:list` 或等效路由检查
- 核心 service 的单元测试
- 鉴权失败、限流触发、路由不存在、上游超时等异常场景
- MySQL 迁移可重复执行
- Redis 不可用时的降级或错误响应

若当前阶段无法运行测试，必须在回复中说明原因和未验证风险。

## 10. MCP 与外部工具规则

默认离线优先，只有在需要最新信息、官方文档或复杂规划时才调用外部服务。

- 规划分解：优先使用 Sequential Thinking。
- 官方文档/API 查询：优先使用 Context7。
- 最新网页信息：优先使用 DuckDuckGo。
- 代码语义检索和符号级修改：优先使用 Serena。

调用外部服务时需要遵守：

- 单轮对话尽量只使用一种外部服务。
- 查询范围最小化，避免上传敏感信息。
- 回复末尾追加“工具调用简报”，说明工具、输入摘要、参数、时间、来源和重试情况。

## 11. 协作输出格式

实现类任务完成后，回复应包含：

- 修改了哪些文件。
- 实现了什么能力。
- 如何验证。
- 尚未覆盖的风险或后续建议。

代码审查任务应优先列出问题，并包含文件路径和行号。

方案讨论任务应包含：

- 推荐方案。
- 可选方案对比。
- 风险和成本。
- 下一步行动。
