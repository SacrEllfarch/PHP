# 安全设计

## 1. 安全目标

- 防止未授权调用。
- 防止请求重放。
- 限制异常流量对上游服务的影响。
- 防止网关被用作任意转发代理。
- 避免敏感信息出现在日志和响应中。

## 2. 鉴权机制

调用方必须提供：

- `X-App-Key`
- `X-Timestamp`
- `X-Nonce`
- `X-Signature`

校验顺序：

1. Header 是否完整。
2. 应用是否存在。
3. 应用是否启用。
4. 时间戳是否在有效窗口内。
5. nonce 是否已使用。
6. 签名是否正确。

签名比较必须使用恒定时间比较，避免时序攻击。

## 3. 防重放

- nonce 需要足够随机。
- Redis 保存 `app_key + nonce`，TTL 与签名窗口一致。
- 重复 nonce 直接返回 `AUTH_NONCE_REPLAYED`。

## 4. 密钥管理

- `app_secret` 只在创建或重置时返回一次。
- 数据库不得保存明文密钥。
- 管理接口不得展示完整密钥。
- 重置密钥后旧密钥应立即失效。

## 5. 转发安全

必须限制：

- 上游地址协议，只允许 `http` 和 `https`。
- 上游域名或 IP 白名单。
- 请求体大小。
- 转发 Header 白名单或黑名单。
- 上游连接超时和读取超时。

禁止转发：

- 管理接口相关 Header。
- 完整认证密钥。
- 内部调试 Header。
- 客户端伪造的网关内部 Header。

## 6. 限流策略

- 默认启用应用级限流。
- 高风险接口可配置更严格的路由级限流。
- Redis 异常时第一阶段建议失败关闭，返回 `DEPENDENCY_UNAVAILABLE`。
- 限流日志需要记录 app_key、route_id、阈值和当前窗口。

## 7. 日志脱敏

必须脱敏：

- `X-Signature`
- `X-App-Key` 可保留，`app_secret` 不得记录。
- `Authorization`
- `Cookie`
- `Set-Cookie`
- 密码、令牌、身份证、手机号等业务敏感字段。

脱敏示例：

```text
abcdef123456 -> abc***456
```

## 8. 管理接口保护

第一阶段可选方案：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 固定管理令牌 | 实现快，适合本地和内网 | 不适合多人权限管理 |
| IP 白名单 | 防护简单直接 | 网络环境变化时维护成本高 |
| 登录 + RBAC | 权限清晰，可审计 | 实现成本较高 |

推荐第一阶段采用“固定管理令牌 + IP 白名单”，后续升级为登录和 RBAC。

## 9. 常见风险

| 风险 | 防护 |
| --- | --- |
| 重放攻击 | timestamp + nonce |
| 签名伪造 | HMAC-SHA256 + 密钥保护 |
| 暴力请求 | Redis 限流 |
| SSRF | 上游白名单 + 协议限制 |
| 日志泄密 | 敏感字段脱敏 |
| 上游拖垮网关 | 超时、最大响应限制、熔断预留 |
