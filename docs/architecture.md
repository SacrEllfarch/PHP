# 系统架构说明

## 1. 架构风格

系统采用单体应用起步、模块化分层设计。这样可以降低第一阶段开发和部署成本，同时保留后续拆分能力。

推荐分层：

```text
Controller -> Middleware -> Service -> Model/Repository -> MySQL/Redis/Upstream
```

各层职责：

| 层级 | 职责 |
| --- | --- |
| Controller | 接收请求、调用服务、返回统一响应 |
| Middleware | TraceId、鉴权、限流、日志、跨域 |
| Service | 业务规则、流程编排、策略判断 |
| Model/Repository | 数据查询、持久化、事务边界 |
| Infrastructure | Redis、HTTP 客户端、配置、日志 |

## 2. 运行时上下文

```text
调用方应用
  -> API 网关
    -> MySQL：应用、路由、策略、日志
    -> Redis：限流计数、nonce 去重、路由缓存
    -> 上游服务：真实业务 API
```

## 3. 核心链路

### 管理链路

```text
管理员 -> 管理接口 -> 参数校验 -> 管理服务 -> MySQL -> 操作日志
```

### 网关链路

```text
调用方 -> 网关入口 -> 鉴权 -> 限流 -> 路由匹配 -> 转发上游 -> 记录日志 -> 返回响应
```

## 4. 模块划分

| 模块 | 说明 |
| --- | --- |
| 应用管理 | 管理 app_key、app_secret、状态和备注 |
| 路由管理 | 管理网关路径、上游地址、匹配类型、超时 |
| 鉴权模块 | 验证身份、签名、时间戳和 nonce |
| 限流模块 | Redis 计数和策略判断 |
| 转发模块 | 构造上游请求、处理响应和异常 |
| 日志模块 | 请求日志、错误日志、管理操作日志 |
| 健康检查 | 检查应用、MySQL、Redis 依赖状态 |

## 5. 扩展点

- 鉴权策略可从 HMAC 扩展到 JWT、OAuth2 或内部令牌。
- 限流算法可从固定窗口升级到滑动窗口或令牌桶。
- 路由来源可从 MySQL 扩展到配置中心或服务发现。
- 日志可从 MySQL 扩展到消息队列、Elasticsearch 或 ClickHouse。
- 管理接口可接入 RBAC 权限模型。

## 6. 关键设计原则

- 主链路保持短路径，避免过多同步外部依赖。
- 管理数据以 MySQL 为准，运行时热点数据可缓存在 Redis。
- 配置变更后需要明确缓存刷新策略。
- 所有跨系统请求必须有超时。
- 网关错误响应不能泄露内部堆栈和上游敏感信息。
