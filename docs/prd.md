## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 简单 API 网关系统 |
| 文档版本 | v0.1 |

## 2. 产品背景

随着后端接口数量增加，调用方直接访问各个上游服务会带来几个问题：

- 调用入口分散，接口地址、鉴权方式和限流规则难以统一管理。
- 上游服务直接暴露给调用方，安全边界不清晰。
- 缺少统一请求日志，排查问题时难以追踪调用链路。
- 缺少基础流量保护，上游服务容易被异常请求拖垮。
- 应用密钥、路由、策略散落在代码或配置中，维护成本逐渐升高。

本系统通过一个轻量 API 网关作为统一入口，先解决“接入、鉴权、限流、转发、记录”这条主链路，为后续扩展服务治理能力打基础。

## 3. 产品定位

本产品是一个面向中小型项目或内部系统的轻量 API 网关，不追求替代 Kong、APISIX、Nginx Plus 等成熟网关。

第一阶段定位：

- 作为内部 API 的统一访问入口。
- 作为调用方应用的认证和管控中心。
- 作为上游服务的基础保护层。
- 作为接口调用日志和问题排查入口。

## 4. 产品目标

### 4.1 业务目标

- 降低新调用方接入成本。
- 统一接口鉴权和限流策略。
- 减少上游服务被异常流量影响的概率。
- 提供可追踪的接口访问日志。
- 为后续管理后台、审计、告警和统计分析预留基础数据。

### 4.2 MVP 目标

MVP 必须完成：

- 应用创建、启停和密钥生成。
- 路由创建、启停和请求匹配。
- HMAC-SHA256 签名鉴权。
- timestamp + nonce 防重放。
- Redis 固定窗口限流。
- HTTP 请求转发。
- 请求日志和错误日志。
- 健康检查。
- 基础管理 API。

### 4.3 成功指标

| 指标 | MVP 目标 |
| --- | --- |
| 接入效率 | 新应用能在 5 分钟内完成密钥创建和首个路由调用 |
| 可用性 | 本地和测试环境主链路可稳定运行 |
| 鉴权准确性 | 错误签名、过期 timestamp、重复 nonce 均能被拦截 |
| 限流准确性 | 超过阈值请求返回 429，且不转发上游 |
| 可追踪性 | 每个网关请求都有 trace_id 和日志记录 |
| 安全性 | 日志和接口响应不泄露 app_secret |

## 5. 用户角色

| 角色 | 核心诉求 | 典型行为 |
| --- | --- | --- |
| 网关管理员 | 管理调用方、路由和策略 | 创建应用、重置密钥、配置路由、启停策略 |
| 调用方开发者 | 安全稳定地调用后端接口 | 获取 app_key、生成签名、请求网关接口 |
| 上游服务开发者 | 保护服务并减少重复鉴权逻辑 | 提供上游地址、确认路由和超时策略 |
| 运维人员 | 快速定位异常和依赖故障 | 查看健康检查、请求日志、错误日志 |

## 6. 使用场景

### 6.1 新应用接入

管理员为调用方创建应用，系统生成 `app_key` 和 `app_secret`。调用方根据签名规则发起请求，网关验证通过后转发到上游服务。

价值：

- 应用身份清晰。
- 密钥可单独停用或重置。
- 调用行为可追踪。

### 6.2 上游服务统一暴露

管理员配置网关路径和上游地址。调用方只访问网关地址，不直接感知真实上游服务地址。

价值：

- 减少上游地址暴露。
- 后续上游迁移时可通过路由配置调整。
- 网关层可统一处理超时和错误响应。

### 6.3 异常流量保护

管理员配置应用级或接口级限流策略。调用方请求超过阈值后，网关返回 429，不再访问上游服务。

价值：

- 防止单个调用方影响整体服务。
- 为上游服务提供基础流量保护。
- 异常请求有日志可查。

### 6.4 故障排查

调用方反馈接口异常时，运维人员可根据 `trace_id` 查询网关日志，确认请求是否通过鉴权、是否命中限流、是否上游超时。

价值：

- 快速区分调用方问题、网关问题和上游问题。
- 降低跨团队排查成本。

## 8. 产品流程

### 8.1 应用接入流程

```text
管理员创建应用
  -> 系统生成 app_key 和 app_secret
  -> 管理员将密钥交付给调用方
  -> 调用方按签名规则请求网关
  -> 网关验证身份并继续处理
```

### 8.2 路由配置流程

```text
管理员创建路由
  -> 配置网关路径、方法、匹配类型、上游地址、超时
  -> 启用路由
  -> 调用方请求网关路径
  -> 网关匹配路由并转发上游
```

### 8.3 网关请求流程

```text
调用方请求
  -> trace_id 初始化
  -> Header 参数校验
  -> 应用状态校验
  -> 签名和防重放校验
  -> 限流判断
  -> 路由匹配
  -> 请求转发
  -> 日志记录
  -> 返回响应
```

## 9. 详细需求

### 9.1 应用管理

#### 9.1.1 创建应用

输入：

- 应用名称
- 备注，可选
- 默认状态，可选，默认启用

输出：

- 应用 ID
- `app_key`
- `app_secret`
- 应用状态

规则：

- 应用名称不能为空。
- `app_key` 必须全局唯一。
- `app_secret` 只在创建和重置时返回一次。
- 数据库不得保存明文 `app_secret`。

验收标准：

- 创建成功后可以在应用列表中查询。
- 创建响应包含一次性明文 `app_secret`。
- 后续查询接口不返回明文 `app_secret`。

#### 9.1.2 启停应用

规则：

- 禁用应用后，所有该应用请求返回 `APP_DISABLED`。
- 启用应用后，应用可重新参与鉴权。

验收标准：

- 禁用状态下，即使签名正确也不能访问网关。
- 操作日志记录启停行为。

#### 9.1.3 重置密钥

规则：

- 重置后旧密钥立即失效。
- 新密钥只返回一次。

验收标准：

- 旧密钥签名请求失败。
- 新密钥签名请求成功。

### 9.2 路由管理

#### 9.2.1 创建路由

输入：

- 路由名称
- HTTP 方法
- 网关路径
- 匹配类型：精确匹配、前缀匹配
- 上游地址
- 优先级
- 超时配置
- 状态

规则：

- 网关路径必须以 `/` 开头。
- 上游地址必须符合协议和白名单要求。
- 禁用路由不参与匹配。

验收标准：

- 启用路由可被请求匹配。
- 禁用路由返回 `ROUTE_NOT_FOUND` 或继续寻找其他匹配路由。

#### 9.2.2 路由匹配

规则：

- 优先匹配 HTTP 方法。
- `ANY` 方法可匹配所有方法。
- 精确匹配优先于前缀匹配。
- 同类匹配按优先级降序选择。

验收标准：

- 相同路径下优先级高的路由先命中。
- 未匹配任何路由时返回 404。

### 9.3 鉴权认证

请求必须包含：

- `X-App-Key`
- `X-Timestamp`
- `X-Nonce`
- `X-Signature`

规则：

- 时间戳默认有效窗口为 300 秒。
- nonce 在窗口期内不可重复。
- 签名使用 HMAC-SHA256。
- 签名比较使用恒定时间比较。

验收标准：

- 缺少认证 Header 返回 `AUTH_REQUIRED`。
- 错误签名返回 `AUTH_SIGNATURE_INVALID`。
- 过期时间戳返回 `AUTH_TIMESTAMP_EXPIRED`。
- 重复 nonce 返回 `AUTH_NONCE_REPLAYED`。

### 9.4 限流

规则：

- 支持应用级限流。
- 支持路由级限流。
- 路由级策略优先于应用级策略。
- 第一阶段使用 Redis 固定窗口计数。

验收标准：

- 未超过阈值的请求可继续转发。
- 超过阈值的请求返回 `RATE_LIMITED`。
- 限流请求不访问上游服务。

### 9.5 请求转发

规则：

- 保留请求方法。
- 透传必要 Header。
- 透传 Query 和 Body。
- 设置连接超时和读取超时。
- 限制请求体大小。
- 不透传敏感内部 Header。

验收标准：

- 上游正常响应时，网关返回上游状态码和响应体。
- 上游连接失败返回 `UPSTREAM_ERROR`。
- 上游超时返回 `UPSTREAM_TIMEOUT`。

### 9.6 日志审计

记录字段：

- trace_id
- app_key
- route_id
- 请求方法
- 请求路径
- 请求 IP
- 上游地址
- 响应状态码
- 错误码
- 耗时
- 创建时间

规则：

- 敏感信息必须脱敏。
- 鉴权失败、限流、上游异常都需要记录。
- 管理操作需要记录操作者、动作和对象。

验收标准：

- 每个网关请求都能通过 trace_id 追踪。
- 日志中不出现完整密钥、令牌和签名。

### 9.7 健康检查

检查项：

- 应用自身状态。
- MySQL 连接状态。
- Redis 连接状态。

验收标准：

- 依赖正常时返回健康。
- 任一关键依赖异常时返回明确状态和错误信息，但不泄露敏感配置。

## 12. 非功能需求

### 12.1 性能

- MVP 阶段单实例应支持基础并发调用，具体指标在压测后确定。
- 网关主链路应避免无必要的多次数据库查询。
- 热点路由和策略可使用 Redis 缓存。

### 12.2 可用性

- 所有外部依赖调用必须设置超时。
- MySQL 或 Redis 异常时返回稳定错误响应。
- 上游异常不能导致网关进程崩溃。

### 12.3 安全

- 管理接口必须受保护。
- app_secret 不得明文落库。
- 日志必须脱敏。
- 上游地址必须受协议和白名单约束。

### 12.4 可维护性

- 按 Controller、Middleware、Service、Model 分层。
- 核心鉴权、限流、路由匹配逻辑需要单元测试。
- 所有错误码集中维护。

### 12.5 可观测性

- 每个请求都有 trace_id。
- 网关处理耗时可记录。
- 鉴权失败、限流、上游异常可分类统计。

## 13. 权限和安全策略

MVP 管理接口推荐使用：

- 固定管理令牌。
- 内网或 IP 白名单限制。
- 管理操作日志。

后续版本再升级：

- 管理员账号。
- 角色权限。
- 操作审批。
- 密钥轮换策略。

## 14. 验收标准

MVP 验收需要覆盖：

- 管理员能创建应用并获得密钥。
- 调用方能用正确签名访问已配置路由。
- 错误签名、过期 timestamp、重复 nonce 被拒绝。
- 超过限流阈值返回 429。
- 未配置路由返回 404。
- 上游超时返回 504。
- 请求日志能记录成功、失败、限流和上游异常。
- 健康检查能展示 MySQL 和 Redis 状态。
- 文档、接口、数据库和实现保持一致。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 需求边界扩大 | MVP 延期 | 明确非目标，先完成主链路 |
| Redis 不稳定 | 限流和 nonce 校验失败 | 明确失败关闭策略并做好健康检查 |
| 路由配置错误 | 请求转发失败或安全风险 | 增加参数校验、上游白名单和操作日志 |
| 日志同步写入影响性能 | 网关响应变慢 | MVP 先同步记录，后续可异步化 |
| app_secret 泄露 | 调用身份被冒用 | 不明文落库、日志脱敏、支持重置密钥 |
| 上游响应过慢 | 网关资源被占用 | 设置连接和读取超时 |
