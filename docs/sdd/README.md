# SDD 工作流

SDD 是本项目的默认开发模式。它的核心思想是先让“要做什么”和“怎样算完成”变清楚，再进入设计和编码。

## 流程

```text
需求澄清 -> 规格编写 -> 技术设计 -> 任务拆分 -> 实现 -> 验证 -> 文档同步
```

## 阶段说明

| 阶段 | 产物 | 目标 |
| --- | --- | --- |
| 需求澄清 | 用户故事、业务规则 | 确认系统要解决的问题 |
| 规格编写 | `requirements.md` | 明确功能边界和验收标准 |
| 技术设计 | `design.md` | 明确架构、数据、接口、异常、安全 |
| 任务拆分 | `tasks.md` | 拆成可执行、可验证的小任务 |
| 实现 | 代码和迁移 | 按任务交付功能 |
| 验证 | 测试和调试记录 | 证明功能符合规格 |
| 文档同步 | 更新 Markdown | 保证知识不过期 |

## 需求写法

需求应使用用户故事和验收标准描述：

```text
作为调用方应用，
我希望通过 App-Key 和签名访问网关，
以便网关能识别调用身份并阻止伪造请求。
```

验收标准推荐使用 `Given / When / Then`：

```text
Given 应用处于启用状态且签名正确
When 调用方请求已配置路由
Then 网关应转发请求并返回上游响应
```

## 设计写法

设计文档至少包含：

- 目标和非目标
- 系统上下文
- 请求处理链路
- 模块职责
- 数据模型
- API 草案
- 错误码
- 安全策略
- 性能与可观测性
- 风险和取舍

## 任务写法

任务必须满足：

- 一个任务只做一类明确事情。
- 每个任务有验收方式。
- 任务之间依赖关系清楚。
- 优先完成可运行主链路，再补扩展能力。
