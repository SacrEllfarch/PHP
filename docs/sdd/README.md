# SDD 工作流

SDD 是本项目的默认开发模式。它的核心思想是以 PRD 作为产品基线，先让“为什么做、为谁做、做到什么程度”变清楚，再把产品目标拆成可设计、可编码、可验证的工程规格。

## 流程

```text
PRD 产品基线 -> SDD 需求规格 -> 技术设计 -> 任务拆分 -> 实现 -> 验证 -> 文档同步
```

## 阶段说明

| 阶段 | 产物 | 目标 |
| --- | --- | --- |
| PRD 产品基线 | `docs/prd.md` | 明确产品背景、目标用户、MVP 范围、成功指标和验收口径 |
| SDD 需求规格 | `requirements.md` | 将 PRD 拆成用户故事、业务规则、功能需求和异常场景 |
| 技术设计 | `design.md` | 将需求映射为架构、数据、接口、错误码、安全和可观测性方案 |
| 任务拆分 | `tasks.md` | 将设计拆成可执行、可验证、可追踪的小任务 |
| 实现 | 代码和迁移 | 按任务交付功能，并保持实现与规格一致 |
| 验证 | 测试和调试记录 | 用自动化测试、接口测试和手工验证证明功能符合规格 |
| 文档同步 | 更新 Markdown | 需求、设计、任务、数据库和安全文档保持同步 |

## 文档层级

```text
PRD：回答为什么做、为谁做、做到什么程度
  -> requirements：回答具体做什么、业务规则是什么、如何验收
    -> design：回答怎么实现、数据如何流动、异常如何处理
      -> tasks：回答谁先做、做哪些文件、怎么验证
```

PRD 不直接替代 SDD 规格。PRD 是产品约束，SDD 是工程契约。任何功能进入开发前，都必须能从 `tasks.md` 追溯到 `design.md`、`requirements.md` 和 `prd.md`。

## 编号规则

为保证可追溯性，需求、设计和任务统一使用编号：

| 类型 | 前缀 | 示例 | 说明 |
| --- | --- | --- | --- |
| 产品目标 | `PG` | `PG-001` | 来自 PRD 的业务目标或 MVP 目标 |
| 用户故事 | `US` | `US-001` | 用户视角的功能诉求 |
| 业务规则 | `BR` | `BR-001` | 必须遵守的业务约束 |
| 功能需求 | `FR` | `FR-001` | 可实现、可测试的功能要求 |
| 非功能需求 | `NFR` | `NFR-001` | 性能、安全、可用性、可维护性要求 |
| 设计决策 | `DD` | `DD-001` | 架构或实现层面的关键决策 |
| 开发任务 | `TASK` | `TASK-001` | 可执行任务 |

## 追踪矩阵

每个规格目录至少维护一张轻量追踪矩阵：

| PRD 条目 | SDD 需求 | 设计章节 | 开发任务 | 验证方式 |
| --- | --- | --- | --- | --- |
| `PG-001` | `FR-001` | `DD-001` | `TASK-001` | 单元测试、接口测试 |

追踪矩阵的价值不是形式化管理，而是避免“PRD 里有、设计没覆盖、任务没实现”的断层。

## 需求写法

需求应从 PRD 中抽取用户角色、使用场景和 MVP 范围，并使用用户故事和验收标准描述：

```text
作为调用方应用，
我希望通过 App-Key 和签名访问网关，
以便网关能识别调用身份并阻止伪造请求。
```

验收标准推荐使用 `Given / When / Then`：

```text
Given 应用处于启用状态且签名正确
When 调用方请求已配置路由
Then 网关应转发请求并返回上游响应
```

## 设计写法

设计文档至少包含：

- PRD 和需求追踪关系
- 目标和非目标
- 系统上下文
- 请求处理链路
- 模块职责
- 数据模型
- API 草案
- 错误码
- 安全策略
- 性能与可观测性
- 风险和取舍

## 任务写法

任务必须满足：

- 一个任务只做一类明确事情。
- 每个任务有验收方式。
- 任务之间依赖关系清楚。
- 优先完成可运行主链路，再补扩展能力。
- 每个任务标注关联的 `FR`、`NFR` 或 `DD` 编号。

## 变更规则

- 产品目标、MVP 范围、成功指标变化：先改 `docs/prd.md`。
- 用户故事、业务规则、验收标准变化：同步改 `requirements.md`。
- 架构、数据、接口、错误码、安全策略变化：同步改 `design.md`。
- 实施顺序、验收命令、任务状态变化：同步改 `tasks.md`。
- 数据库字段变化：同步改 `docs/database.md`。
- 鉴权、签名、限流、日志脱敏变化：同步改 `docs/security.md`。
